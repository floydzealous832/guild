# Expected output

Captured snapshots from a reference run of this example. Generated by running
the example in the sandbox project (`guild-example-02-quest-decomposition`) and
saving output from:

- `guild quest list -p guild-example-02-quest-decomposition --status done` → `quest-list-final.txt`
- `guild lore list -p guild-example-02-quest-decomposition` (filtered to current) → `lore-catalog.txt`
- `guild quest bounties -p guild-example-02-quest-decomposition` (brief section) → `brief.txt`

## What you should see

After a successful run:

**quest-list-final.txt** — 5 quests all with status `done`:
- parent quest (`Plan a rate limiter`)
- sub-quest: algorithm recommendation
- sub-quest: storage backend recommendation
- sub-quest: pitfalls list
- sub-quest: observability signal

QUEST IDs will differ in your run (guild assigns IDs globally across the database).

**lore-catalog.txt** — 5 new entries created by the run:

| Kind | Content | Reason |
|---|---|---|
| `decision` | algorithm choice (e.g. token bucket) | subagent committed to a specific choice |
| `decision` | storage backend (e.g. Redis) | subagent committed to a specific choice |
| `decision` | observability signal (e.g. throttle_ratio gauge) | subagent committed to a specific choice |
| **`research`** | **3 pitfalls to avoid** | **list-shape content: cataloging failure modes, not committing to a single path** |
| `decision` | parent synthesis (integrated design) | parent committed to the overall design |

**Note on the pitfalls entry being `research`, not `decision`** — this is intentional and reproducible across independent runs. The pitfalls subagent produces a list of domain failure modes with generic mitigations. That's reference knowledge to inform implementation, not a project-specific commitment. Research is the semantically correct kind. See cross-project LORE-243 for the full finding.

**brief.txt** — one-paragraph handoff written by the parent after synthesis. Names the integrated design, the 4 sub-decisions that informed it, and the implementation starting point. Substantive enough that a cold agent picking up this project next session has real orientation.

## Note on LORE IDs

All LORE IDs in the snapshots are from the reference run and will differ in yours. Guild assigns IDs sequentially across the whole `~/.guild/*.db`. The content, count (5 new entries), and kind distribution (4 decisions + 1 research) are what matter.
